### Fix 8: Convert function declarations to variable expressions for reassignment

### Learnings
- In TypeScript (and strict JavaScript), you cannot reassign a variable declared with `function` statement - it creates a function declaration which is read-only

- The semicolons after function expressions are important (`;`) since they're now variable assignments, not function declarations

### Fix 9: Wrap boolean expressions with Number() for arithmetic operations
- **Error**: `The left-hand side of an arithmetic operation must be of type 'any', 'number', 'bigint' or an enum type.` (TS2362/TS2363)
- **Location**: `src/main/msx/cpu/CPU.ts` (57 occurrences throughout the file)
- **Solution**: Wrapped boolean comparison expressions with `Number()` when used in arithmetic operations (bitwise OR and left shift)
- **Patterns fixed**:
  - `((A === 0) << nZ)` → `(Number(A === 0) << nZ)` (and similar for res, B, val)
  - `((B + C !== 0) << nPV)` → `(Number(B + C !== 0) << nPV)`
  - `(((res & 0x0f) === 0) << nH)` → `(Number((res & 0x0f) === 0) << nH)` (and similar nested patterns)
  - `| (A > 0x99);` → `| Number(A > 0x99);` (end-of-expression booleans)
- **Result**: Fixed 57 TS2362 errors and 4 TS2363 errors in CPU.ts
- **Remaining errors**: 558
- **Commit**: 9492fe3d

### Learnings
- In Z80/R800 CPU emulation, flag registers are computed using bitwise operations where boolean conditions set specific bits
- JavaScript implicitly converts `true` to `1` and `false` to `0` in arithmetic contexts, but TypeScript requires explicit conversion
- `Number(boolExpr)` returns `1` for `true` and `0` for `false`, making it ideal for flag bit calculations
- The pattern `(condition << bitPosition)` sets a specific bit in a flags register - this is a common idiom in CPU emulators
- There are two contexts where booleans appear in arithmetic:
  1. Left side of shift: `((res === 0) << nZ)` - boolean shifted to correct bit position
  2. End of bitwise OR chain: `| (A > 0x99);` - boolean ORed directly into result
- When using sed for bulk replacements, apply patterns from most specific (longest/nested) to least specific to avoid corrupting nested patterns
