## Consolidated Learnings

### Global Declarations
- Use `.d.ts` files in tsconfig includes path to declare global variables (`declare var name: type;`)
- The codebase uses namespace pattern where `wmsx.ClassName = {...}` assigns modules to global object
- `wmsx` (lowercase) is the class namespace, `WMSX` (uppercase) is the configuration namespace

### TypeScript Configuration
- `lib` compiler option controls available type definitions, separate from `target` (output format)
- Adding `"ES2015"` enables modern APIs like `Number.parseFloat` without changing runtime behavior
- Runtime (browser) must still support features; ES2015 is widely supported

### Type Conversions
- `String.match()` returns string arrays even for digit patterns - use `Number()` for explicit conversion
- JavaScript allows implicit coercion in bitwise operations, TypeScript enforces explicit conversion
- `Number(boolean)` returns 1 for true, 0 for false - ideal for CPU flag bit calculations
- Pattern `(condition << bitPosition)` sets specific bits in registers (common in CPU emulators)
- Variables used in bitwise operations (`<<`, `|`, `&`, `^`) should be declared as numbers, not booleans
- XOR operator `^` requires numeric operands - convert boolean expressions with `Number()` or declare as numbers

### Function Declarations
- `function name() {}` creates read-only function declaration that cannot be reassigned
- Use `var name = function() {};` for reassignable functions
- Function expressions behave identically at runtime with same scope rules
- Used for providing different implementations based on configuration (e.g., R800 vs Z80 timing)

### Loop Variables
- `for...in` loops always iterate over property keys as strings, even for array indices
- String type from `for...in` persists throughout function scope due to `var` hoisting
- Use different variable names for string keys (`for...in`) vs numeric indices (`for` loops)

### Object Initialization
- TypeScript infers type from initial object literal and doesn't allow adding properties afterward
- Initialize objects with all properties at once: `var obj = { prop1: val1, prop2: val2 }`
- For conditional properties use: `prop: condition ? value : undefined`
- For truly optional properties use spread operator: `...(condition ? { prop: value } : {})`
- Compute dependent values as local variables before creating object literal
- This pattern is critical for CPU instruction objects and state serialization
- Use object spread `{ ...source }` to copy objects instead of `for...in` loops for better type inference
- Pattern `var obj = {}; for (k in src) obj[k] = src[k];` should be replaced with `var obj = { ...src };`
- For arrays of objects with many optional properties, define an explicit type to avoid complex union inference
- Pattern: `type ItemType = { prop1: type; prop2?: type; prop3?: type }; var items: ItemType[] = [...]`
- TypeScript infers array element types as union of all shapes, which can exclude properties not in all elements

### Optional Parameters
- Use `param?` for optional parameters where `undefined` behavior is significant
- Use `param = defaultValue` when you want a specific default instead of `undefined`
- `param?` preserves nuanced behavior (e.g., `undefined` vs `false` vs `true`)
- `param = false` is appropriate for force/override flags that default to normal operation
- When removing unused parameters is cleaner than adding arguments to all call sites
- Default parameters preserve backward compatibility with existing call sites
- Use `param = undefined` to explicitly document that parameter is optional and can be omitted at call sites
- Use union types like `boolean | number` for parameters that accept both boolean and numeric flags (0/1), common in network protocols

### Event Handlers
- Event handler properties (e.g., `onerror`, `ontimeout`) have specific type signatures expecting event parameters
- Don't manually invoke event handlers without proper arguments - call the underlying logic directly instead
- Pattern: Instead of `element.onerror()` (missing event param), use direct function call or pass a proper event
- Event handler signatures should not be changed to make parameters optional when they're browser-defined contracts

### Type Corrections
- Variables should be declared with types matching their usage throughout the code
- Use `number | null` for variables that hold numeric values or "absent" state
- Both `false` and `null` are falsy, but `null` documents intent better for non-boolean uses
- Explicit type annotations (`var name: type`) clearly document variable intent

### Deprecated APIs and Custom Properties
- When dealing with deprecated browser APIs (like `applicationCache`), add custom type declarations
- Extend built-in interfaces like `Window` or `Document` in `.d.ts` files via declaration merging
- Optional properties (`prop?`) indicate features may not exist in modern environments
- Vendor-prefixed APIs (`moz*`, `webkit*`) follow same pattern for legacy browser support
- For custom properties added to DOM elements at runtime, extend appropriate DOM interfaces
- Extend both `Element` and `HTMLElement` when properties are used on elements from different sources
- `document.elementFromPoint()` returns `Element`, while `document.createElement()` returns specific element types
- Extended `EventTarget` can include DOM-specific properties like `parentElement` used in event handlers
- Custom properties on browser objects (`File`, `Error`, etc.) follow same interface extension pattern

### Control Flow Analysis
- TypeScript performs type narrowing based on conditional checks (`if`, `else if`, `in` operator)
- `else if` creates mutually exclusive branches that can over-narrow types to `never` in edge cases
- When checking for independent optional properties, use separate `if` statements instead of `else if`
- Pattern: `if ("prop1" in obj) {...} if ("prop2" in obj) {...}` works better than `else if` for optional features
- This is especially important for vendor-prefixed browser APIs where multiple prefixes can exist independently

### Consistency Patterns
- When multiple related functions share similar patterns, keep signatures consistent
- Parameterization (passing values vs accessing globals) improves testability and flexibility
- Common patterns: force parameters, extended state, palette parameters, infinite command flags
- Video chip modules (VDP, V9990) show consistent architecture across emulator subsystems

### sed and Bulk Replacements
- Apply patterns from most specific (longest/nested) to least specific to avoid corrupting code
- Nested patterns like `((res & 0x0f) === 0)` must be matched before simpler patterns

### Type Assertions
- Use `as Type` when you have more specific knowledge than TypeScript's inference
- FileReader methods: `readAsArrayBuffer()` returns ArrayBuffer, `readAsText()` returns string
- TypeScript types `event.target.result` conservatively as union of all possible return types
- Type assertions clarify runtime guarantees without changing behavior

### MSX Architecture Context
- VDP: TMS9918-derived video chip family (V9918, V9938, V9958) for standard MSX
- V9990: Yamaha's advanced video chip for MSX TurboR GT (higher resolution/color depth)
- CPU emulation: Z80/R800 processors with flag registers computed via bitwise operations
- Video modes: Graphics 6 (Screen 7) is 256x212 with 512 colors, pixels doubled horizontally
- Save states: Support normal and extended modes (extended includes debug state)
- Commands: Finite (pixel-count based) vs infinite (CPU-driven) graphics operations
